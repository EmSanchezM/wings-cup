# Design: foundation — Nuxt 4.4 + Supabase Bootstrap

> Technical design for the `foundation` slice. Architecture decisions, full SQL (schema + RLS + triggers), auth sequences, env contract, testing wiring, and Vercel deploy shape. Locked decisions: A1 (cloud-only Supabase), B1 (score cap 15), C1 (Vercel cron), D1 (7-day invite expiry). Delivery: `auto-chain` 3 stacked PRs.

This design lifts and refines the 13 architecture decisions from `proposal.md` (referenced inline as `[P-#]`). New material is marked `[NEW]`.

## 4. Database schema design (`00001_schema.sql`)

All 7 tables ship in this single migration. Why one file: foundation is greenfield; splitting tables across files adds nothing and complicates the FK ordering. New migrations (slice 2+) append to history.

### 4.1 `profiles` — extends `auth.users`

```sql
CREATE TABLE profiles (
  id              UUID PRIMARY KEY REFERENCES auth.users(id) ON DELETE CASCADE,
  display_name    TEXT NOT NULL,
  avatar_url      TEXT,
  auth_provider   TEXT NOT NULL CHECK (auth_provider IN ('google','magic_link')),
  is_guest        BOOLEAN NOT NULL DEFAULT TRUE,
  is_super_admin  BOOLEAN NOT NULL DEFAULT FALSE,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
CREATE INDEX idx_profiles_is_super_admin ON profiles(is_super_admin) WHERE is_super_admin = TRUE;
```
Why: `id` is also the FK to `auth.users` — one row per Supabase user. `updated_at` deviates from spec [proposal §Spec deviations #3] for cache + audit. Partial index on super-admin keeps it cheap (almost all rows are FALSE).

### 4.2 `rooms`

```sql
CREATE TABLE rooms (
  id             UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name           TEXT NOT NULL,
  invite_code    TEXT NOT NULL UNIQUE,
  status         TEXT NOT NULL DEFAULT 'active' CHECK (status IN ('active','closed')),
  scoring_rules  JSONB NOT NULL DEFAULT '{"exact_score":5,"correct_goal_diff":3,"correct_result":1}'::jsonb,
  created_by     UUID NOT NULL REFERENCES profiles(id) ON DELETE RESTRICT,
  created_at     TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
CREATE INDEX idx_rooms_invite_code ON rooms(invite_code);
CREATE INDEX idx_rooms_created_by ON rooms(created_by);
```
Why `ON DELETE RESTRICT` on `created_by`: deleting a profile that owns rooms must fail loudly — slice 5 will offer ownership transfer.

### 4.3 `room_members` — composite PK + REPLICA IDENTITY FULL

```sql
CREATE TABLE room_members (
  room_id       UUID NOT NULL REFERENCES rooms(id) ON DELETE CASCADE,
  user_id       UUID NOT NULL REFERENCES profiles(id) ON DELETE CASCADE,
  role          TEXT NOT NULL DEFAULT 'member' CHECK (role IN ('owner','member')),
  total_points  INTEGER NOT NULL DEFAULT 0,
  joined_at     TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  PRIMARY KEY (room_id, user_id)
);
ALTER TABLE room_members REPLICA IDENTITY FULL;          -- [P-11] slice 4 realtime
CREATE INDEX idx_room_members_user_id ON room_members(user_id);
CREATE INDEX idx_room_members_room_points ON room_members(room_id, total_points DESC);
```
Why FULL: realtime UPDATE payloads need OLD-row values to filter `room_id` client-side without an extra round trip. Setting it now avoids a slice-4 hot-table ALTER. The `(room_id, total_points DESC)` composite supports the leaderboard query in slice 4.

### 4.4 `matches`

```sql
CREATE TABLE matches (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  external_id   TEXT UNIQUE,                              -- API-Football fixture id
  home_team     TEXT NOT NULL,
  away_team     TEXT NOT NULL,
  kickoff_at    TIMESTAMPTZ NOT NULL,
  status        TEXT NOT NULL DEFAULT 'scheduled'
                  CHECK (status IN ('scheduled','live','finished','postponed','cancelled')),
  home_score    INTEGER CHECK (home_score IS NULL OR home_score >= 0),
  away_score    INTEGER CHECK (away_score IS NULL OR away_score >= 0),
  group_name    TEXT,                                     -- 'A','B'… or NULL for knockout
  stage         TEXT NOT NULL DEFAULT 'group'
                  CHECK (stage IN ('group','round_of_16','quarter','semi','final','third_place'))
);
CREATE INDEX idx_matches_kickoff_at ON matches(kickoff_at);
CREATE INDEX idx_matches_status ON matches(status);
CREATE UNIQUE INDEX idx_matches_external_id ON matches(external_id) WHERE external_id IS NOT NULL;
```
Why partial UNIQUE: lets us seed manual matches with NULL `external_id` while still enforcing API-Football idempotency.

### 4.5 `predictions` — score cap 15 [Locked Decision B]

```sql
CREATE TABLE predictions (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  room_id         UUID NOT NULL REFERENCES rooms(id) ON DELETE CASCADE,
  user_id         UUID NOT NULL REFERENCES profiles(id) ON DELETE CASCADE,
  match_id        UUID NOT NULL REFERENCES matches(id) ON DELETE CASCADE,
  predicted_home  INTEGER NOT NULL CHECK (predicted_home BETWEEN 0 AND 15),
  predicted_away  INTEGER NOT NULL CHECK (predicted_away BETWEEN 0 AND 15),
  points_awarded  INTEGER NOT NULL DEFAULT 0,
  locked_at       TIMESTAMPTZ,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  UNIQUE (room_id, user_id, match_id)
);
CREATE INDEX idx_predictions_room_user ON predictions(room_id, user_id);
CREATE INDEX idx_predictions_match ON predictions(match_id);
CREATE INDEX idx_predictions_unlocked ON predictions(match_id) WHERE locked_at IS NULL;
```
Why the partial `unlocked` index: the cron job `lock_started_predictions()` queries `WHERE locked_at IS NULL AND kickoff_at <= NOW()` every minute; a partial index keeps it fast even when the table grows.

### 4.6 `invitations` — 7-day default [Locked Decision D]

```sql
CREATE TABLE invitations (
  id                 UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  room_id            UUID NOT NULL REFERENCES rooms(id) ON DELETE CASCADE,
  token              TEXT NOT NULL UNIQUE,
  email              TEXT,                          -- optional pre-fill
  created_by         UUID NOT NULL REFERENCES profiles(id) ON DELETE RESTRICT,
  used_by_user_id    UUID REFERENCES profiles(id) ON DELETE SET NULL,
  expires_at         TIMESTAMPTZ NOT NULL DEFAULT (NOW() + INTERVAL '7 days'),
  created_at         TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  used_at            TIMESTAMPTZ
);
CREATE INDEX idx_invitations_token ON invitations(token);
CREATE INDEX idx_invitations_room_id ON invitations(room_id);
```

### 4.7 `audit_log`

```sql
CREATE TABLE audit_log (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  admin_id     UUID NOT NULL REFERENCES profiles(id) ON DELETE RESTRICT,
  action       TEXT NOT NULL,
  target_type  TEXT NOT NULL,                       -- 'room','user','match',…
  target_id    UUID,
  before_value JSONB,
  after_value  JSONB,
  created_at   TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
CREATE INDEX idx_audit_log_admin_created ON audit_log(admin_id, created_at DESC);
```

## 5. RLS policy design (`00002_rls.sql`)

`ENABLE ROW LEVEL SECURITY` on all 7 tables. `service_role` bypasses RLS by Supabase default — server-side cron/admin code uses the service role key explicitly. **No table ships without RLS** — closes the spec gap on `matches`, `invitations`, `audit_log`, `profiles` [P-#13, deviations #5, #6, #7].

### 5.1 `profiles`

```sql
ALTER TABLE profiles ENABLE ROW LEVEL SECURITY;

-- SELECT: own row, OR (display_name, avatar_url) of users in shared rooms
-- is_super_admin column-level lock: server re-validates via serverSupabaseUser()
CREATE POLICY profiles_select_self ON profiles FOR SELECT
  USING (id = auth.uid());

CREATE POLICY profiles_select_shared_room ON profiles FOR SELECT
  USING (
    EXISTS (
      SELECT 1 FROM room_members me
      JOIN room_members other ON me.room_id = other.room_id
      WHERE me.user_id = auth.uid() AND other.user_id = profiles.id
    )
  );

-- INSERT denied to clients; only handle_new_user (SECURITY DEFINER trigger) writes here
CREATE POLICY profiles_insert_none ON profiles FOR INSERT WITH CHECK (false);

-- UPDATE: own row only; super-admin column protected by trigger (5.2)
CREATE POLICY profiles_update_self ON profiles FOR UPDATE
  USING (id = auth.uid()) WITH CHECK (id = auth.uid());

CREATE POLICY profiles_delete_none ON profiles FOR DELETE USING (false);
```

**`is_super_admin` re-validation pattern** — RLS protects the row but never trust the SELECTed value. Every server endpoint that gates on super-admin MUST:

```ts
// server/api/admin/*.ts
const user = await serverSupabaseUser(event)
const supabase = await serverSupabaseServiceClient(event)
const { data } = await supabase.from('profiles').select('is_super_admin').eq('id', user.id).single()
if (!data?.is_super_admin) throw createError({ statusCode: 403 })
```
Service-role read bypasses RLS, so we get the real column value, not a client-stripped one.

### 5.2 `profiles` super-admin column lock (trigger, lives in `00003_triggers.sql`)

```sql
CREATE FUNCTION lock_super_admin_column() RETURNS TRIGGER LANGUAGE plpgsql AS $$
BEGIN
  IF NEW.is_super_admin IS DISTINCT FROM OLD.is_super_admin THEN
    -- Allow only when current_setting indicates we're running as service_role
    IF current_setting('request.jwt.claims', true)::jsonb->>'role' <> 'service_role' THEN
      RAISE EXCEPTION 'is_super_admin can only be modified by service role';
    END IF;
  END IF;
  RETURN NEW;
END $$;
CREATE TRIGGER profiles_lock_super_admin BEFORE UPDATE ON profiles
  FOR EACH ROW EXECUTE FUNCTION lock_super_admin_column();
```

### 5.6 `predictions`

```sql
ALTER TABLE predictions ENABLE ROW LEVEL SECURITY;

CREATE POLICY pred_select_room_members ON predictions FOR SELECT
  USING (EXISTS (
    SELECT 1 FROM room_members WHERE room_id = predictions.room_id AND user_id = auth.uid()
  ));

CREATE POLICY pred_insert_own_before_kickoff ON predictions FOR INSERT
  WITH CHECK (
    user_id = auth.uid()
    AND EXISTS (SELECT 1 FROM matches WHERE matches.id = predictions.match_id AND matches.kickoff_at > NOW())
    -- NOTE: kickoff_at is TIMESTAMPTZ, NOW() returns TIMESTAMPTZ — UTC-safe.
    -- Symmetric with lock_started_predictions() which uses kickoff_at <= NOW().
  );

CREATE POLICY pred_update_own_unlocked ON predictions FOR UPDATE
  USING (
    user_id = auth.uid() AND locked_at IS NULL
    AND EXISTS (SELECT 1 FROM matches WHERE matches.id = predictions.match_id AND matches.kickoff_at > NOW())
  ) WITH CHECK (user_id = auth.uid());

CREATE POLICY pred_delete_own_unlocked ON predictions FOR DELETE
  USING (user_id = auth.uid() AND locked_at IS NULL);
```

## 6. Trigger and function design (`00003_triggers.sql`)

### 6.1 `calculate_points()` — runs after match finalizes

```sql
CREATE OR REPLACE FUNCTION calculate_points() RETURNS TRIGGER
LANGUAGE plpgsql SECURITY DEFINER AS $$
DECLARE
  rules JSONB;
  pts   INTEGER;
  pred  RECORD;
BEGIN
  IF NEW.status <> 'finished' OR OLD.status = 'finished' THEN
    RETURN NEW;
  END IF;
  IF NEW.home_score IS NULL OR NEW.away_score IS NULL THEN
    RETURN NEW;
  END IF;

  FOR pred IN
    SELECT p.id, p.user_id, p.room_id, p.predicted_home, p.predicted_away, r.scoring_rules
    FROM predictions p JOIN rooms r ON r.id = p.room_id
    WHERE p.match_id = NEW.id AND p.locked_at IS NOT NULL
  LOOP
    rules := pred.scoring_rules;
    -- Priority order (DOCUMENTED): exact_score > correct_goal_diff > correct_result > 0.
    -- A prediction can only earn ONE bucket — the highest matching one wins.
    pts := CASE
      WHEN pred.predicted_home = NEW.home_score AND pred.predicted_away = NEW.away_score
        THEN (rules->>'exact_score')::int
      WHEN (pred.predicted_home - pred.predicted_away) = (NEW.home_score - NEW.away_score)
        THEN (rules->>'correct_goal_diff')::int
      WHEN sign(pred.predicted_home - pred.predicted_away) = sign(NEW.home_score - NEW.away_score)
        THEN (rules->>'correct_result')::int
      ELSE 0
    END;

    UPDATE predictions SET points_awarded = pts WHERE id = pred.id;
    UPDATE room_members
      SET total_points = total_points + pts
      WHERE room_id = pred.room_id AND user_id = pred.user_id;
  END LOOP;
  RETURN NEW;
END $$;

CREATE TRIGGER matches_calculate_points AFTER UPDATE OF status, home_score, away_score ON matches
  FOR EACH ROW EXECUTE FUNCTION calculate_points();
```

### 6.2 `lock_started_predictions()` — called by Vercel cron

```sql
CREATE OR REPLACE FUNCTION lock_started_predictions() RETURNS INTEGER
LANGUAGE plpgsql SECURITY DEFINER AS $$
DECLARE
  affected INTEGER;
BEGIN
  -- Caller: /api/cron/lock-predictions (slice 3) authenticates with CRON_SECRET
  -- and invokes via service-role client — bypasses RLS.
  -- TIMESTAMPTZ + NOW() = UTC-safe; symmetric with predictions INSERT policy.
  UPDATE predictions p
    SET locked_at = NOW()
    FROM matches m
    WHERE p.match_id = m.id
      AND p.locked_at IS NULL
      AND m.kickoff_at <= NOW();
  GET DIAGNOSTICS affected = ROW_COUNT;
  RETURN affected;
END $$;
```

### 6.3 `handle_new_user()` — profile-on-signup

```sql
CREATE OR REPLACE FUNCTION handle_new_user() RETURNS TRIGGER
LANGUAGE plpgsql SECURITY DEFINER AS $$
DECLARE
  provider TEXT;
BEGIN
  -- OAuth populates app_metadata.provider; magic link signup leaves it null.
  provider := COALESCE(NEW.raw_app_meta_data->>'provider', 'magic_link');

  INSERT INTO public.profiles (id, display_name, avatar_url, auth_provider, is_guest)
  VALUES (
    NEW.id,
    COALESCE(NEW.raw_user_meta_data->>'display_name',
             NEW.raw_user_meta_data->>'full_name',
             NEW.raw_user_meta_data->>'name',
             split_part(NEW.email, '@', 1)),
    NEW.raw_user_meta_data->>'avatar_url',
    provider,
    provider = 'magic_link'             -- magic-link signup = guest by default
  );
  RETURN NEW;
END $$;

CREATE TRIGGER auth_users_create_profile AFTER INSERT ON auth.users
  FOR EACH ROW EXECUTE FUNCTION handle_new_user();
```

## 8. `redirectOptions` route map

| Route            | Public/Protected | Behavior |
|------------------|------------------|----------|
| `/`              | Public (exclude) | Landing page; no redirect |
| `/auth/login`    | Public (exclude) | Login UI; matches `redirectOptions.login` |
| `/auth/confirm`  | Public (exclude) | OAuth + magic-link callback; exchanges code, sets cookie, redirects to saved path |
| `/auth/*` (other)| Public (exclude) | Future: signup, link-identity confirm |
| `/join/{code}`   | Public (exclude) | Magic-link entry for invited guests; allows anonymous form submit |
| `/rooms`         | **Protected** (include) | Module redirects to `/auth/login?redirect=/rooms` if no session |
| `/rooms/{id}`    | **Protected** (include) | Same; `saveRedirectToCookie` returns user here post-login |
| `/admin`         | **Protected** (include) | Module enforces login; server route additionally re-validates `is_super_admin` |
| `/admin/*`       | **Protected** (include) | Same |

`saveRedirectToCookie: true` means a logged-out user hitting `/rooms/abc` gets sent to `/auth/login`, then back to `/rooms/abc` after auth. Cookie name: `sb-redirect-path` (module default).

## 9. Environment variable contract

| Variable                       | Scope         | Example pattern                                | Used by |
|--------------------------------|---------------|------------------------------------------------|---------|
| `SUPABASE_URL`                 | Public (build) | `https://xxxx.supabase.co`                     | `@nuxtjs/supabase` reads directly |
| `SUPABASE_ANON_KEY`            | Public (build) | `eyJhbGciOi...` (long JWT, anon role)          | `@nuxtjs/supabase` reads directly |
| `NUXT_SUPABASE_SERVICE_KEY`    | **Server only** | `eyJhbGciOi...` (service_role JWT)             | `runtimeConfig.supabaseServiceKey` → server-side `serverSupabaseServiceClient()` |
| `NUXT_CRON_SECRET`             | **Server only** | random 64-char hex (`openssl rand -hex 32`)    | `runtimeConfig.cronSecret` → `/api/cron/*` header check (slice 3) |
| `NUXT_API_FOOTBALL_KEY`        | **Server only** | API-Football dashboard key                     | `runtimeConfig.apiFootballKey` → match sync (slice 3) |

**Leak-prevention pattern** [P-10]:
- Service-role key NEVER goes into `runtimeConfig.public.*` and NEVER prefixed with `NUXT_PUBLIC_`.
- Nuxt convention: `NUXT_*` (without `PUBLIC_`) maps to top-level `runtimeConfig.*` which is server-only.
- CI smoke test (slice 5 hardening; foundation only documents): `grep -r "service_role" .output/public && exit 1`.
- `.env.example` contains every key with placeholder values; never commit `.env`.

## 12. Migration ordering

```
supabase/migrations/
├── 00001_schema.sql      ← all 7 tables, indexes, REPLICA IDENTITY FULL
├── 00002_rls.sql         ← ENABLE RLS on all 7 + every policy from §5
└── 00003_triggers.sql    ← all functions + triggers from §6 (calculate_points, lock_started_predictions, handle_new_user, set_updated_at, lock_super_admin_column)

supabase/seed.sql         ← (optional) manual super-admin bootstrap; not auto-applied in prod
```

**Why this order**: tables before RLS (RLS needs tables), RLS before triggers (triggers may need policy context for SECURITY DEFINER reasoning), triggers last (depend on schema and may depend on RLS being enabled to test the bypass behavior).

**Dev-prod connection model** [Locked Decision A1]:
- `supabase link --project-ref <DEV_REF>` against a free-tier project named `wings-cup-dev`.
- Apply migrations: `supabase db push`.
- Reset dev DB: `supabase db reset --linked`.
- Production: separate project (`wings-cup-prod`); same `db push` flow run from CI on merge to `main`.
- `seed.sql` is local-only (runs via `supabase db reset` but not `db push` to prod).

## 14. Risks and design-time mitigations

| Risk | Sev | Design-level mitigation |
|------|-----|------------------------|
| Spec gap: `matches`/`invitations`/`audit_log` shipped without RLS | High | Closed in `00002_rls.sql` (§5.5, §5.7, §5.8); foundation success criteria asserts SELECT-as-anon returns 0 |
| `is_super_admin` leak via `profiles` SELECT | High | RLS policy returns own row + only `display_name,avatar_url` of shared-room peers; server re-validates super-admin via service-role read (§5.1, §5.2) |
| `total_points` mutated by clients | High | Both column-grant REVOKE and SECURITY DEFINER trigger (§5.4 + §6.1) |
| `NUXT_SUPABASE_SERVICE_KEY` in client bundle | Critical | `runtimeConfig.supabaseServiceKey` server-only; NEVER `runtimeConfig.public.*`; CI grep planned (slice 5) |
| `@nuxt/test-utils` v4 `beforeAll` pitfall | Medium | Smoke test uses v4 pattern; README §Testing documents it |
| `shadcn-nuxt` instability | Low | Pin exact version in `package.json`; manual copy fallback path documented |
| Realtime forgotten until slice 4 | Medium | `REPLICA IDENTITY FULL` set in `00001_schema.sql` |
| `pg_cron` unavailable on free tier | Low | Vercel cron strategy locked (§11) |
| Profile trigger fires on phantom users (SAML, anon) | Low | `COALESCE(...,'magic_link')` defensive default; only INSERT path adds profile |
| Tailwind v4 OKLCH muscle-memory regression | Low | Tokens in `app/assets/css/tailwind.css` with comments; README links Tailwind v4 migration guide |
| Migration applied to wrong project (dev vs prod) | High | `supabase link` confirms project name in CLI prompt; CI deploy uses separate `SUPABASE_PROJECT_REF` env per environment |
| Cron caller spoofing | Medium | `CRON_SECRET` header check pattern documented (§11); slice 3 implements |
| Same email used for OAuth + magic link causes duplicate users | Medium | Supabase identity-linking handles this; guest→registered flow (§7.3) reuses same `user_id` |
